# 矛盾类型速查表

将《矛盾论》中的抽象分类对应到实际工作场景中，帮助快速识别和定性矛盾。

## 按性质分类

### 对抗性矛盾（根本冲突，不可调和，必须选边）

| 场景 | 对立面A | 对立面B | 处理方式 |
|------|--------|--------|---------|
| 技术选型 | 方案A（互斥架构） | 方案B（互斥架构） | 果断选择，不做折中 |
| 安全漏洞 | 修复漏洞 | 保持现状不修 | 安全优先，不容妥协 |
| 数据一致性 | 强一致 | 最终一致 | 根据业务需求二选一 |
| 破坏性变更 | 推进重构 | 维持向后兼容 | 明确取舍，设定迁移期 |

### 非对抗性矛盾（共同目标下的分歧，可以调和）

| 场景 | 对立面A | 对立面B | 处理方式 |
|------|--------|--------|---------|
| 速度 vs 质量 | 快速交付 | 充分测试 | 关键路径严测，非关键快迭代 |
| 技术债 vs 功能 | 偿还技术债 | 开发新功能 | 每个迭代分配比例 |
| 性能 vs 可读性 | 极致优化 | 清晰代码 | 热路径优化，其余优先可读 |
| 自研 vs 引入 | 自建方案 | 使用三方库 | 核心能力自研，辅助引入 |

## 按地位分类

### 判断主要矛盾的三个标准

1. **因果检验**：解决了它，其他矛盾是否随之缓解？
2. **根源检验**：它是否是其他矛盾产生的根源？
3. **阶段检验**：它是否处于当前阶段的中心位置？

### 常见的主次矛盾识别模式

| 表面问题（次要矛盾） | 根本问题（主要矛盾） |
|-------------------|-------------------|
| 频繁线上故障 | 缺乏自动化测试体系 |
| 开发效率低 | 架构耦合度过高 |
| 团队协作冲突 | 职责边界不清晰 |
| 需求频繁变更 | 需求确认流程缺失 |
| 部署失败率高 | CI/CD 流水线不完善 |

## 矛盾转化的常见信号

| 信号 | 可能的转化 |
|------|----------|
| 被搁置的技术债开始频繁引发故障 | 次要→主要 |
| 非对抗性的方案分歧升级为团队对立 | 非对抗→对抗 |
| 原来的性能瓶颈通过扩容暂时解决 | 主要→次要（暂时） |
| 外部依赖方发生重大变更 | 新矛盾可能上升为主要 |
